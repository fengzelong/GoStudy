# 企业应用迭代规划

这份规划用于把当前企业应用骨架继续推进成更完整的后端示例。每一阶段都保持可运行、可测试，不影响原有学习示例。

## 当前基线

- 已有 `cmd/server` 统一入口。
- 已有 Gin 路由、统一响应、请求 ID、CORS、请求日志和鉴权中间件。
- 已有用户注册、登录、任务创建、任务查询和任务完成接口。
- 已有内存仓储和 GORM/MySQL 仓储切换。
- 已有服务层、路由层、仓储工厂和鉴权基础测试。
- 已有 `scripts/check.ps1` 提交前检查脚本。

## 第一阶段：业务可用性

目标：让用户和任务模块更接近真实业务系统。

| 任务 | 说明 | 验收方式 |
| --- | --- | --- |
| 用户资料查询 | 新增当前用户详情接口，例如 `GET /api/v1/me` | 路由测试覆盖未登录、已登录两种情况 |
| 任务归属隔离 | 当前用户只能查看和修改自己的任务 | 路由测试覆盖跨用户访问被拒绝 |
| 列表分页 | 用户列表、任务列表支持 `page`、`page_size` | 服务层和路由层测试覆盖默认值、边界值 |
| 统一错误码 | 将业务错误码从 HTTP 状态码中拆出 | 文档列出错误码，测试断言响应结构 |

## 第二阶段：外部基础设施

目标：把已有中间件示例接入企业骨架。

| 任务 | 说明 | 验收方式 |
| --- | --- | --- |
| Redis 缓存 | 缓存用户详情或任务列表，支持 `APP_CACHE=memory/redis/off` | 无 Redis 时默认关闭，单元测试覆盖缓存接口 |
| RabbitMQ 事件 | 任务创建、完成时发布事件，支持 `APP_MQ=off/rabbitmq` | 默认关闭，不影响本地测试 |
| 数据库健康检查 | `/health` 增加 MySQL、Redis、MQ 检查状态 | memory/off 模式返回跳过状态 |
| 配置校验 | 启动时检查 storage/cache/mq 配置组合 | 非法配置启动失败并返回明确错误 |

## 第三阶段：认证与权限

目标：从演示鉴权升级到更安全、更清晰的企业权限模型。

| 任务 | 说明 | 验收方式 |
| --- | --- | --- |
| JWT 或统一认证 | 将演示 Token 替换为 JWT 或认证服务适配层 | Token 测试覆盖签名、过期、篡改 |
| 密码算法升级 | 将 SHA256 演示实现替换为 bcrypt 或 argon2 | 注册、登录测试保持通过 |
| 角色权限 | 增加管理员、普通用户角色 | 路由测试覆盖权限允许和拒绝 |
| 审计日志 | 记录登录、任务状态变更等关键操作 | 单元测试覆盖审计事件构造 |

## 第四阶段：工程化质量

目标：补齐企业项目常见的维护能力。

| 任务 | 说明 | 验收方式 |
| --- | --- | --- |
| OpenAPI 文档 | 输出接口文档或示例请求集合 | README 链接文档入口 |
| 集成测试 | 使用可选 MySQL 环境变量运行真实仓储测试 | 默认跳过，无环境时不影响 `go test ./...` |
| Docker Compose | 提供 MySQL、Redis、RabbitMQ 本地依赖编排 | 文档说明启动和清理方式 |
| CI 检查说明 | 将 `scripts/check.ps1` 作为 CI 门禁示例 | 文档列出 CI 执行命令 |

## 推荐推进顺序

1. 先做第一阶段的任务归属隔离和分页，让现有接口更像真实业务。
2. 再接 Redis 缓存，保持默认关闭，避免增加本地学习成本。
3. 接 RabbitMQ 事件，把现有 `gorabbitmq` 示例迁移成可选基础设施。
4. 升级认证和密码算法，明确当前演示实现的边界。
5. 最后补 Docker Compose、OpenAPI 和集成测试。

## 约束

- 默认运行路径必须不依赖 MySQL、Redis、RabbitMQ。
- 新增外部服务能力必须支持关闭或内存替代。
- 每个阶段至少补充单元测试或路由测试。
- 文档、注释和测试提示继续使用中文。
- 不把原有学习示例目录改造成重型项目，企业骨架保持在 `cmd/server` 和 `internal` 下演进。
