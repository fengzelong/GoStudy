# AGENTS 指南

这份文档用于约定本项目的编码氛围和协作方式。目标不是把学习仓库变成沉重的工程规范，而是让每次改动都更清楚、更容易验证，也更适合持续积累。

## 基本原则

- 先理解现有示例，再动手修改。
- 保持改动小而明确，避免顺手重构无关目录。
- 文档、注释和测试提示统一使用中文。
- 示例代码优先表达学习目的，不为了抽象而抽象。
- 能跑测试就跑测试，不能跑时说明原因。
- 不提交构建产物、日志、trace、上传文件和本地缓存。

## 编码习惯

- Go 代码统一使用 `gofmt` 格式化。
- 新增示例时，目录名尽量直接表达主题。
- 同一目录下避免放多个互相冲突的 `main()` 入口；客户端、服务端等入口可以拆成子目录。
- 公共类型和协议结构尽量放到独立包中，方便客户端和服务端共用。
- 注释只解释意图、边界或容易误解的地方，不重复代码表面含义。

## 测试与验证

根模块常用验证命令：

```sh
go test ./...
```

如果本机 Go 缓存目录没有写权限，可以临时把缓存放到仓库内：

```sh
$env:GOMODCACHE="$PWD\.tmp_gomodcache"
$env:GOPATH="$PWD\.tmp_gopath"
go test ./...
```

VS Code 配置已经为测试和调试任务设置了仓库内临时缓存目录。

## 文档约定

- 新增主题示例时，同步更新 `README.md` 的目录说明。
- 新增运行方式复杂的示例时，在对应目录补充简短说明。
- 如果示例依赖 MySQL、Redis、RabbitMQ 等本地服务，需要写明前置条件。
- 如果某个测试是为了演示失败场景，不要让它破坏默认的 `go test ./...`。

## AI 协作约定

与 AI 助手一起改代码时，推荐按下面节奏推进：

1. 先让 AI 查看目录结构、模块和当前 Git 状态。
2. 明确本次目标，只改和目标相关的文件。
3. 修改前说明准备动哪些地方。
4. 修改后运行格式化和测试。
5. 最后说明改动内容、验证结果和遗留问题。

如果发现工作区已有未提交改动，默认认为这些改动是用户或其他工具留下的，不要随意回滚。

## 提交流程建议

提交前检查：

```sh
git status --short
go test ./...
```

提交信息建议使用简洁中文，例如：

```sh
git commit -m "整理项目结构与开发配置"
```
